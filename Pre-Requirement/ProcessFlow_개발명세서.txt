=======================================================================
  ProcessFlow — AI 개발 명세서 (Claude Code용)
  버전: v1.3  |  작성일: 2025.03  |  최종수정: 2026.03
=======================================================================

이 문서는 ProcessFlow 웹 애플리케이션의 전체 개발 명세서입니다.
React SPA로 구현하며, 텍스트 데이터는 LocalStorage, 이미지 데이터는
IndexedDB에 분리 저장합니다. Claude API 호출은 Netlify Functions 프록시를
경유하여 API 키를 서버사이드에서 보관합니다.
Word 내보내기는 브라우저에서 docx-js 라이브러리로 직접 생성합니다.

[v1.3 변경 사항]
  - 이미지 저장소: LocalStorage Base64 → IndexedDB Blob 분리
  - AI API 호출: 브라우저 직접 호출 → Netlify Functions 프록시 경유
  - Claude 모델명: 컴포넌트 하드코딩 → constants.js CLAUDE_MODEL 상수로 분리
  - Image 타입: data(Base64) 필드 → id(IndexedDB key) 필드로 변경


=======================================================================
1. 프로젝트 개요
=======================================================================

서비스명: ProcessFlow
목적: 사용자가 업무 흐름을 입력하면 자동으로 프로세스를 시각화하고
      단계별 상세 매뉴얼(Word 문서)을 생성해주는 웹 플랫폼
대상: SAP 기반 업무를 수행하는 기업 내 팀(CO, FI, MM 등)

핵심 결과물:
  1. 수영 레인 다이어그램 (부서별 업무 흐름 시각화)
  2. a→b→c 일렬 흐름도 (단계 순서 한눈에 파악)
  3. Word 매뉴얼 (그룹 단위, 단계별 상세 + 이미지 포함)


=======================================================================
2. 기술 스택
=======================================================================

프레임워크: React 18 (Hooks 기반)
스타일링: Inline CSS (CSS-in-JS, 외부 라이브러리 없음)
다이어그램: SVG 직접 구현 (CSS Grid + SVG 오버레이 방식)
텍스트 저장: LocalStorage (key: "processflow_v1", JSON.stringify)
이미지 저장: IndexedDB (DB명: "processflow_images", 수백MB 가능)
            - idb 라이브러리 사용 (경량 IndexedDB wrapper, ~3KB)
Word 생성: docx 라이브러리 (브라우저에서 직접 Blob 생성)
폰트: 'Malgun Gothic', '맑은 고딕', 'Noto Sans KR', sans-serif
AI API: Anthropic Claude API — Netlify Functions 프록시 경유 (필수)
        모델명: constants.js의 CLAUDE_MODEL 상수 참조
        프론트엔드 엔드포인트: POST /.netlify/functions/claude
Word 이미지: IndexedDB Blob → ArrayBuffer 변환 후 ImageRun으로 실제 삽입
배포: Netlify (React 빌드 + Functions 통합 배포)
빌드 도구: Vite 5.x (Create React App 대신 사용, 빌드 속도 우수)

4. 데이터 구조 (TypeScript 타입 기준)
=======================================================================

-- 4.1 최상위 구조 --

type AppData = Department[]

-- 4.2 Department (LV1) --

type Department = {
  id: string           // "d1", "d"+Date.now()
  name: string         // "CO팀", "재무팀"
  icon: string         // 이름 첫 글자 또는 이모지
  color: string        // hex, 사이드바 아이콘 배경색
  groups: Group[]
}

-- 4.3 Group (LV2) --

type Group = {
  id: string           // "g1", "g"+Date.now()
  name: string         // "마감 프로세스"
  processes: Process[]
  // 주의: code 필드 없음 (UI에서 번호 표시 안 함)
}

-- 4.4 Process (LV3) --

type Process = {
  id: string           // "p1", "p"+Date.now()
  name: string         // "물류마감 Check 프로세스"
  dept: string         // "CO팀" — Word 내보내기 및 표시용
  owner: string        // "김재무" — 프로세스 담당자
  module: string       // "CO", "FI", "MM"
  updatedAt: string    // "2025.03"
  description: string  // 프로세스 간단 설명
  steps: Step[]
}

-- 4.5 Step (단계) --

type Step = {
  id: string           // "s1", "s"+Date.now()
  title: string        // "물류 미결 현황 조회"
  screenName: string   // "MB52" — T-Code 또는 프로그램명
  dept: string         // "CO팀" — 이 단계 담당 부서 (수영 레인 분류 기준)
  pt: string           // "30min" — 처리 시간
  logic: string        // 업무 로직 (줄바꿈 \n 포함 가능)
  images: Image[]      // 참고 이미지 배열
  warning: string      // 주의사항 (없으면 빈 문자열 "")
}

-- 4.6 Image --

type Image = {
  id: string           // IndexedDB 저장 키 ("img_"+Date.now())
  name: string         // 원본 파일명 "mb52_화면.png"
}

// 주의: data(Base64) 필드 제거됨. 이미지 실데이터는 IndexedDB에서 id로 조회.

-- 4.7 LocalStorage 저장 --

키: "processflow_v1"
값: JSON.stringify(Department[])
    (Image.id만 포함, 이미지 Blob 데이터는 포함되지 않음)

저장 시점:
  - 프로세스 생성/삭제
  - 그룹 생성/삭제
  - 부서 생성/삭제
  - 단계 추가/수정/삭제

용량 처리:
  - Blob.size로 용량 계산
  - 3MB 초과 시 console.warn (이미지 없는 텍스트만이므로 여유 충분)
  - QuotaExceededError 발생 시 사용자에게 alert

-- 4.8 IndexedDB 저장 --

DB명: "processflow_images"
버전: 1
스토어명: "images"
키: id (string, "img_"+Date.now())

저장 레코드:
  { id: string, blob: Blob, name: string, createdAt: number }

저장 시점:
  - StepModal에서 이미지 첨부 시 즉시 저장 (저장 버튼 누르기 전)
  - 실제로는 onSave 시점에 저장 후 id 반환 → Step.images에 기록

삭제 시점:
  - 단계 내 이미지 × 버튼 클릭 시 해당 id 삭제
  - 단계 삭제 시 step.images의 모든 id 일괄 삭제
  - 프로세스/그룹/부서 삭제 시 하위 모든 이미지 id 수집 후 일괄 삭제

5. 앱 계층 구조 (LV1 → LV2 → LV3)
=======================================================================

LV1 — 부서 목록 화면
  - URL 개념: 부서 선택 상태
  - 내용: 해당 부서의 Group 카드 목록
  - 기능: 그룹 추가 버튼, 부서 삭제 버튼(카드별)

LV2 — 프로세스 그룹 화면
  - URL 개념: 부서 + 그룹 선택 상태
  - 내용: 해당 그룹의 Process 카드 목록
  - 기능:
      * 그룹 단위 Word 내보내기 버튼 (헤더)
      * 그룹 삭제 버튼 (헤더)
      * 프로세스 추가 버튼 (헤더)
      * 각 카드: 상세 보기 → / 🗑 삭제 / ▼ 수영 레인 펼치기
  - 내보내기 범위: 해당 그룹 내 모든 프로세스 전체를 1개 Word 파일로

LV3 — 세부 프로세스 상세 화면
  - URL 개념: 부서 + 그룹 + 프로세스 선택 상태
  - 내용: 프로세스 메타 + 전체 흐름도(일렬) + 단계별 상세
  - 기능:
      * Word 내보내기 버튼 없음 (LV2에서만 가능)
      * 단계 추가 버튼 (우상단 + 하단)
      * 각 단계 헤더: ✏️ 수정 / 🗑 삭제
      * 단계 본문: 읽기 전용 표시 (수정 불가, 펜 버튼으로만 수정)

7. 컴포넌트 명세
=======================================================================

-- 7.1 LinearFlow --
props: { steps: Step[] }
용도: LV3 전체 흐름도 (a→b→c)
구현:
  - 가로 방향 flex, overflowX: auto
  - 각 노드: border 2px solid C.blue, radius:8
  - 노드 내용: 단계명(11px bold) + screenName(9px blue) + PT(10px gray)
  - 단계 번호 뱃지: 우상단 absolute, circle, C.blue 배경
  - 화살표: SVG polygon (→)

-- 7.2 SwimLane --
props: { steps: Step[] }
용도: LV2 카드 ▼ 확장 시 부서별 수영 레인
구현:
  - CSS Grid (80px 부서 컬럼 + 나머지)
  - 헤더 행: C.navy 배경
  - 레인 행: 짝수 white / 홀수 gray100
  - 각 부서 컬러: DEPT_COLORS = [navy, blue, "#059669", "#d97706", "#7c3aed"]
  - 노드: 해당 레인에만 표시, 다른 레인은 빈 div
  - 정방향 화살표: SVG, 같은 레인이면 부서 색, 레인 전환이면 gray 점선
  - 분기/반려: 현재 MVP에서 미구현 (v2 예정)

-- 7.3 StepCard --
props: { step: Step, index: number, onEdit: fn, onDelete: fn }
용도: LV3 단계별 상세 카드
구현:
  헤더 (항상 표시):
    - 번호 badge (28px, navy→blue 활성)
    - 단계명 (13px bold) + "화면: screenName · dept · PT: pt"
    - [✏️] 수정 버튼 → StepModal 오픈
    - [🗑] 삭제 버튼 → confirm 후 onDelete 호출
    - [▲/▼] 펼치기 토글
  
  본문 (▼ 펼쳤을 때):
    배경: white, borderTop
    읽기 전용 표시 (input/textarea 없음)
    
    Logic 섹션:
      라벨: "Logic" (11px bold blue)
      내용: gray100 배경 박스, 13px, lineHeight 1.8, whiteSpace: pre-line
    
    참고 이미지 섹션 (images.length > 0 인 경우만):
      라벨: "📎 참고 이미지  N개" (11px bold blue)
      이미지 목록: 
        - 썸네일 120px × 80px, radius:8, objectFit: cover
        - 각 이미지 아래 파일명 표시
        - flex wrap 배치
    
    주의사항 섹션 (warning 값 있는 경우만):
      배경: #FFFBEB, 테두리: #FDE68A

저장 함수 (별도 분리 — v2에서 Firebase로 교체 용이하게):
  function saveToStorage(data) {
    try {
      const json = JSON.stringify(data)
      const sizeKB = Math.round(new Blob([json]).size / 1024)
      if (sizeKB > 3000) console.warn("Storage size warning:", sizeKB, "KB")
      localStorage.setItem("processflow_v1", json)
      return { ok: true, sizeKB }
    } catch (e) {
      if (e.name === "QuotaExceededError") return { ok: false, error: "storage_full" }
      return { ok: false, error: e.message }
    }
  }

  function loadFromStorage() {
    try {
      const raw = localStorage.getItem("processflow_v1")
      return raw ? JSON.parse(raw) : null
    } catch { return null }
  }

  초기 데이터: useState(() => loadFromStorage() || SAMPLE_DATA)
  저장 시점: setData 래핑 함수에서 saveToStorage 자동 호출

IndexedDB 함수 (imageDB.js — idb 라이브러리 사용):

  async function openImageDB() {
    return openDB("processflow_images", 1, {
      upgrade(db) { db.createObjectStore("images", { keyPath: "id" }) }
    })
  }

  async function saveImage(id, blob, name) {
    const db = await openImageDB()
    await db.put("images", { id, blob, name, createdAt: Date.now() })
  }

  async function loadImage(id) {
    const db = await openImageDB()
    return db.get("images", id)   // { id, blob, name, createdAt }
  }

  async function deleteImage(id) {
    const db = await openImageDB()
    await db.delete("images", id)
  }

  async function deleteImages(ids) {
    const db = await openImageDB()
    const tx = db.transaction("images", "readwrite")
    await Promise.all(ids.map(id => tx.store.delete(id)))
    await tx.done
  }

확인 후 실행 순서:
  1. 해당 Group 하위 전체 이미지 id 수집
       const imageIds = group.processes
         .flatMap(p => p.steps)
         .flatMap(s => s.images)
         .map(img => img.id)
  2. deleteImages(imageIds) — IndexedDB에서 일괄 삭제
  3. 해당 Department.groups에서 제거
  4. saveToStorage() — LocalStorage 저장
사이드 이펙트: selGroup, selProc null로 초기화

-- 11.3 프로세스(LV3) 삭제 --
트리거: LV2 카드 내 [🗑] 버튼
팝업 메시지: "{프로세스명}을 삭제할까요?"
확인 후 실행 순서:
  1. 해당 Process 하위 전체 이미지 id 수집
       const imageIds = process.steps
         .flatMap(s => s.images)
         .map(img => img.id)
  2. deleteImages(imageIds) — IndexedDB에서 일괄 삭제
  3. 해당 Group.processes에서 제거
  4. saveToStorage() — LocalStorage 저장

-- 11.4 단계 삭제 --
트리거: LV3 StepCard 헤더 [🗑] 버튼
동작: window.confirm("이 단계를 삭제할까요?") → 확인 시 삭제
확인 후 실행 순서:
  1. step.images의 모든 id 수집
       const imageIds = step.images.map(img => img.id)
  2. deleteImages(imageIds) — IndexedDB에서 일괄 삭제
  3. 해당 Process.steps에서 제거
  4. saveToStorage() — LocalStorage 저장
별도 모달 없이 confirm 사용

-- 11.5 단계 내 이미지 개별 삭제 --
트리거: StepModal 이미지 썸네일의 [×] 버튼
동작:
  1. deleteImage(image.id) — IndexedDB에서 즉시 삭제
  2. 모달 내 임시 images 배열에서 해당 항목 제거
  3. 모달 저장 시 step.images 업데이트 → saveToStorage()
주의: 모달에서 [취소] 클릭 시 이미 IndexedDB에서는 삭제된 상태
      → 취소 시 복구 불가 (허용 가능 수준, 별도 롤백 불필요)

-- 11.6 삭제 실행 헬퍼 함수 (imageDB.js) --
  // 빈 배열 전달 시 아무 동작 안 함 (안전 처리)
  async function deleteImages(ids) {
    if (!ids || ids.length === 0) return
    const db = await openImageDB()
    const tx = db.transaction("images", "readwrite")
    await Promise.all(ids.map(id => tx.store.delete(id)))
    await tx.done
  }

16. Netlify Functions 명세
=======================================================================

-- 16.1 구조 --

파일 위치: netlify/functions/claude.js
엔드포인트: POST /.netlify/functions/claude
역할: 프론트엔드 ↔ Anthropic API 사이 프록시
      (API 키를 클라이언트에 노출하지 않기 위한 서버사이드 레이어)

-- 16.2 요청/응답 형식 --

요청 (프론트엔드 → Function):
  Method: POST
  Headers: Content-Type: application/json
  Body:
    {
      userInput: string,   // 사용자가 입력한 업무 흐름 텍스트
      dept: string         // 담당 부서명
    }

응답 (Function → 프론트엔드):
  성공 (200):
    { steps: Step[] }      // 파싱 완료된 단계 배열

  실패:
    400: { error: "입력값 누락" }
    500: { error: "AI 응답 오류 또는 JSON 파싱 실패" }

-- 16.3 Function 내부 처리 흐름 --

  1. 요청 body 파싱 및 userInput, dept 유효성 검사
  2. Anthropic SDK로 Claude API 호출
     - 모델: process.env.CLAUDE_MODEL || "claude-sonnet-4-20250514"
     - API 키: process.env.ANTHROPIC_API_KEY (Netlify 환경변수)
  3. 응답 텍스트에서 JSON 배열 추출 (정규식으로 ```json ... ``` 제거)
  4. JSON.parse() 후 Step[] 구조 검증
  5. { steps } 반환

-- 16.4 netlify.toml 설정 --

  [build]
    command = "npm run build"
    publish = "build"
    functions = "netlify/functions"

  [[redirects]]
    from = "/*"
    to = "/index.html"
    status = 200

-- 16.5 로컬 개발 환경 --

  - netlify dev 명령으로 React 앱 + Functions 동시 실행
  - 로컬 환경변수: .env.local 파일에 ANTHROPIC_API_KEY 설정
    (이 파일은 .gitignore에 반드시 포함)
  - 로컬 Functions 엔드포인트: http://localhost:8888/.netlify/functions/claude
