ProcessFlow 소프트웨어 설계 명세서 v1.1

1.1 배경 및 목적
현업 담당자들은 업무 인수인계, 신입 온보딩, 감사 대응 등 다양한 상황에서 SOP 문서와 업무 흐름도를 필요로 한다. 현재 대부분의 조직은 Visio, PowerPoint로 수작업 제작하는 방식에 의존하고 있어 작성 시간이 길고 버전 관리가 어렵다.
ProcessFlow는 사용자가 자연어로 업무 흐름을 입력하면 AI가 자동으로 프로세스를 구조화하고, 수영 레인 다이어그램 + 단계별 상세 매뉴얼을 동시에 생성해주는 웹 기반 플랫폼이다.
핵심적으로 다음 두 가지 결과물을 하나의 도구에서 만들어내는 것이 목표다.

플로우차트 — 전체 흐름을 한눈에 파악하는 수영 레인 다이어그램 (분기·반려 경로 포함)
상세 매뉴얼 — 단계별 설명 + 이미지(스크린샷) + 주의사항이 담긴 업무 문서


1.3 프로젝트 범위
v1 범위 (현재)

웹 기반 SPA 구현

2. 시스템 아키텍처
2.1 전체 구성 (v1)
v1은 프론트엔드 단독 구성이다. 별도 서버 없이 브라우저 내에서 모든 것이 동작한다.
[사용자 브라우저]
    │
    ├── React SPA (UI 렌더링, 상태 관리)
    │       │
    │       ├── LocalStorage (프로세스 데이터 + 이미지 Base64 저장)
    │       │
    │       └── Anthropic Claude API (새 프로세스 생성 시 AI 호출)
    │
    └── 내보내기: html2canvas / jsPDF → 파일 다운로드
v2 전환 시 LocalStorage 저장/불러오기 함수만 Firebase SDK 호출로 교체하면 나머지 코드 변경 없이 클라우드 저장으로 전환 가능하도록 저장 로직을 별도 함수(saveToStorage, loadFromStorage)로 분리해두었다.

9. 오류 복구 시나리오
9.1 설계 원칙
- 오류가 발생해도 사용자가 입력한 데이터는 절대 초기화하지 않는다
- 복구 행동은 항상 사용자가 선택한다 (자동 삭제 금지)
- 오류 메시지는 원인과 복구 방법을 함께 제시한다

9.2 AI 응답 JSON 파싱 실패

원인: Claude가 마크다운 코드블록 포함, 필드 누락, 텍스트 혼입 등 비정형 응답 반환

복구 전략 (3단계 순서대로 시도)
  1차 - 자동 보정 (사용자에게 노출 없음)
    - 정규식으로 ```json ... ``` 블록 추출 시도
    - 불완전 JSON 자동 보정 후 재파싱

  2차 - 재요청 버튼 노출
    - "AI 응답을 파싱하지 못했습니다. 다시 시도할까요?" 모달
    - 동일 입력값으로 Claude API 1회 재호출

  3차 - 수동 입력 전환
    - "직접 단계를 입력하시겠어요?" -> Step1 입력 폼으로 이동
    - AI 생성 없이 빈 프로세스 생성 허용

핵심 제약: 프로세스명·부서·흐름 텍스트 입력값은 어느 단계에서도 초기화 금지

9.3 브랜치 구조 오류

원인: fromStepId / toStepId가 참조하는 Step이 삭제됨, 잘못된 데이터 로드

복구 전략
  로드 시 유효성 검사 (자동, 조용히 처리)
    - branches 배열 순회 -> 존재하지 않는 stepId 참조 자동 제거
    - 사용자에게 별도 알림 없음

  편집 중 Step 삭제 시
    - confirm 모달: "이 단계를 삭제하면 연결된 분기 경로 N개도 함께 삭제됩니다"
    - 사용자 확인 후에만 삭제 진행

  다이어그램 렌더링 오류 시
    - SVG 렌더링 try-catch -> 오류 난 브랜치만 점선으로 표시
    - "일부 경로를 표시할 수 없습니다 [경로 초기화]" 버튼 노출

9.4 이미지 첨부 변환 실패

원인: 파일 용량 초과, 지원하지 않는 포맷(HEIC 등), FileReader 오류

복구 전략
  사전 차단 (업로드 전 검사)
    - 파일 크기 5MB 초과 -> "파일이 너무 큽니다 (최대 5MB)"
    - 지원 포맷: jpg, png, gif, webp만 허용, 그 외 포맷 차단

  변환 실패 시
    - 해당 파일만 실패 처리, 나머지 파일은 정상 첨부 유지
    - "OO.heic 파일은 지원하지 않습니다. JPG/PNG로 변환 후 첨부해주세요"

  부분 성공 시
    - "3개 중 2개 첨부 성공" 상태 표시
    - 실패 파일만 재시도 버튼 개별 노출

9.5 저장 중 Quota 초과

원인: LocalStorage 5MB 한계 도달 (이미지 누적)

복구 전략 (경고 -> 차단 -> 복구 순서)
  4.5MB 도달 시 (경고 단계)
    - 상단 배너: "저장 공간이 거의 찼습니다 (92%). 이미지를 줄여주세요"
    - 저장은 계속 허용

  5MB 초과 시 (차단 단계)
    - 모달: "저장 공간이 가득 찼습니다"
    - 복구 옵션 A: 이미지 첨부 많은 프로세스 목록 표시 -> 선택 삭제 유도
    - 복구 옵션 B: 전체 데이터 JSON으로 백업 후 일부 삭제

  공통 제약
    - 저장 실패 시에도 현재 메모리 상태(작업 중 데이터)는 유지
    - "탭을 닫기 전까지 현재 작업은 유지됩니다" 안내 문구 표시

9.6 Word 생성 실패

원인: docx 라이브러리 내부 오류, 이미지 데이터 손상, 메모리 부족

복구 전략
  1차 - 이미지 제외 재시도 (자동 제안)
    - 첫 실패 시 "이미지 없이 Word 파일을 생성할까요?" 모달 표시
    - 사용자 확인 후 이미지 제외 버전 자동 생성

  2차 - 텍스트 파일 폴백
    - Word 생성도 실패 시 "단계별 텍스트(.txt)로 저장" 대안 제공

  공통
    - 오류 정보 [클립보드 복사] 버튼 제공 (관리자 보고용)

9.7 구현 우선순위

  필수 (v1 MVP)
    - JSON 파싱 자동 보정 + 수동 전환 (AI 핵심 기능)
    - Quota 경고 배너 + 차단 모달
    - 이미지 사전 차단 (포맷·용량 검사)

  권장 (안정성)
    - 브랜치 dangling 자동 정리 (로드 시)
    - Word 이미지 제외 재시도

  추후 (v2)
    - 단계별 .txt 폴백
